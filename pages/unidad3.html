🎯 Introducción a la Optimización

La optimización busca mejorar el código generado sin alterar su comportamiento observable. El objetivo puede ser:

⏱️ Velocidad
  • Reducir el número de instrucciones
  • Evitar accesos innecesarios a memoria
  • Aprovechar la caché del procesador
💾 Memoria
  • Reducir el número de temporales
  • Minimizar el uso de registros
  • Reducir el tamaño del binario
⚠️Regla fundamental: la optimización nunca debe cambiar el resultado del programa. Solo cambia la forma en que se obtiene ese resultado.
📌 3.1 Tipos de Optimización
3.1.1 Optimización Local

Se aplica dentro de un bloque básico: una secuencia de instrucciones sin saltos ni etiquetas intermedias. Es la más simple y siempre segura.

a) Eliminación de subexpresiones comunes (CSE)

/* Antes */ t1 = a * b t2 = a * b ← repetición innecesaria t3 = t1 + t2 /* Después */ t1 = a * b t3 = t1 + t1 ← reutiliza t1, ahorra una multiplicación

b) Propagación de constantes

/* Antes */ x = 5 y = x + 3 ← x es constante conocida /* Después */ x = 5 y = 8 ← calculado en tiempo de compilación

c) Plegado de constantes (Constant Folding)

/* Antes */ t1 = 2 * 3 ← ambos operandos son constantes /* Después */ t1 = 6 ← calculado en compilación, no en ejecución

d) Eliminación de código muerto

/* Antes */ x = 5 x = 10 ← la asignación anterior nunca se usa print(x) /* Después */ x = 10 print(x)
3.1.2 Optimización de Ciclos

Los bucles son el objetivo más importante: un código que se ejecuta 10,000 veces tiene 10,000 veces más impacto si es optimizado.

a) Movimiento de código invariante — código que produce el mismo resultado en cada iteración se mueve fuera del ciclo:

/* Antes */ for i = 1 to n: t = a * b ← 'a' y 'b' no cambian en el ciclo arr[i] = t + i /* Después */ t = a * b ← se calcula UNA sola vez for i = 1 to n: arr[i] = t + i

b) Reducción de fuerza — reemplaza operaciones costosas por equivalentes más baratas:

/* Antes (multiplicación en cada iteración) */ for i = 0 to n-1: t = i * 4 ← multiplicación es costosa /* Después (suma en cada iteración) */ t = 0 for i = 0 to n-1: ... ← usa t t = t + 4 ← suma es más rápida

c) Eliminación de variables de inducción — variables que solo cambian linealmente respecto al contador:

/* Antes */ for i = 0 to n: j = i * 8 + base A[j] = ... /* Después */ j = base for i = 0 to n: A[j] = ... j = j + 8 ← j ya no depende de i, se actualiza con suma
3.1.3 Optimización Global

Analiza el flujo de control entre múltiples bloques básicos. Requiere construir el grafo de flujo de control (CFG).

  • Propagación de copias global: si x = y y x no cambia, reemplazar usos de x por y en todos los bloques alcanzables
  • Eliminación de código muerto global: eliminar definiciones de variables que nunca se usan en ningún camino de ejecución
  • CSE global: eliminar subexpresiones comunes entre diferentes bloques básicos
Grafo de flujo de control (CFG): [Bloque 1: x = a + b] | ┌────┴────┐ ↓ ↓ [Bloque 2] [Bloque 3] [y = x * 2] [z = x + 1] └────┬────┘ ↓ [Bloque 4: print(y,z)]

📌 3.2 Costos
3.2.1 Costo de Ejecución

Cada instrucción tiene un costo en ciclos de CPU, accesos a memoria y uso de registros:

Gestión de registros: el compilador asigna variables a registros para evitar accesos a memoria. Cuando los registros se agotan, se produce un spill (desbordamiento): la variable se guarda temporalmente en la pila.

Gestión de pila: cada llamada a función crea un marco de activación con variables locales, parámetros y dirección de retorno. La pila crece y decrece conforme se llaman y retornan funciones.

3.2.2 Criterios para Mejorar el Código
  • Minimizar instrucciones: menos instrucciones = menos ciclos de reloj
  • Favorecer registros sobre memoria: los accesos a RAM son 100× más lentos
  • Evitar divisiones: reemplazar por multiplicaciones por el recíproco cuando sea posible
  • Desenrollar ciclos (loop unrolling): duplicar el cuerpo del ciclo reduce la sobrecarga del contador
  • Vectorización: aprovechar instrucciones SIMD (SSE, AVX) para procesar múltiples datos a la vez
  • Inline de funciones: sustituir llamadas a funciones pequeñas por su cuerpo directo
/* Loop unrolling: antes */ for i = 0 to 7: a[i] = a[i] * 2 /* Loop unrolling: después (desenrollado ×4) */ a[0] = a[0] * 2; a[1] = a[1] * 2 a[2] = a[2] * 2; a[3] = a[3] * 2 a[4] = a[4] * 2; a[5] = a[5] * 2 a[6] = a[6] * 2; a[7] = a[7] * 2 /* Elimina 7 comparaciones y 7 saltos del ciclo */
3.2.3 Herramientas para Análisis del Flujo de Datos

El análisis de flujo de datos recopila información sobre cómo se usan los valores a través del programa:

Grafo de Flujo de Control (CFG)
Los nodos son bloques básicos; los arcos representan posibles transferencias de control (saltos, llamadas). Es la base de todo análisis global.
Reaching Definitions (Definiciones que alcanzan)
Para cada punto del programa, determina qué definiciones de variables pueden haber llegado hasta ese punto sin ser sobreescritas. Se calcula con los conjuntos GEN y KILL por bloque.
Live Variables (Variables vivas)
Una variable está viva en un punto si su valor actual puede ser usado en algún camino futuro. Información crítica para la asignación de registros.
Available Expressions (Expresiones disponibles)
Una expresión está disponible si ya fue calculada y los operandos no han cambiado desde entonces. Permite CSE global.
/* Ejemplo de cálculo de GEN/KILL para reaching definitions */ Bloque B: { d1: x = a + b ← GEN: {d1} d2: y = c * d ← GEN: {d1, d2} d3: x = y + 1 ← KILL: {d1} (nueva def de x) ← GEN: {d2, d3} } KILL(B) = todas las definiciones de x excepto d3 GEN(B) = {d2, d3}
🧠 Conclusión

La optimización es el arte de hacer que el compilador genere código que corra más rápido y use menos recursos. Las técnicas van desde simples transformaciones locales (plegado de constantes, eliminación de código muerto) hasta análisis globales sofisticados (flujo de datos, asignación de registros). El compilador debe equilibrar el tiempo de compilación con la calidad del código generado.
