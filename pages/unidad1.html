🎯 Introducción al Análisis Semántico

El análisis semántico es la tercera fase del proceso de compilación. Después de que el analizador léxico tokeniza el código y el analizador sintáctico valida su estructura gramatical, el analizador semántico verifica que el programa tenga coherencia lógica y de tipos.

💡Un programa puede ser sintácticamente correcto pero semánticamente inválido. Ejemplo: int x = "hola"; tiene sintaxis correcta pero es un error semántico de tipos.
  • Verificar compatibilidad de tipos en expresiones y asignaciones
  • Comprobar que los identificadores sean declarados antes de usarse
  • Validar parámetros en llamadas a funciones
  • Construir la tabla de símbolos con atributos de cada variable
  • Preparar la información para la generación de código intermedio
    • 📌 1.1 Árboles de Expresiones

    Un árbol de expresión es una representación jerárquica de una expresión matemática o lógica, donde:

    • Los nodos internos son operadores (+, -, *, /)
    • Las hojas son operandos (variables, constantes)
    • La estructura refleja la precedencia de operadores automáticamente

    Ejemplo: La expresión a + b * c - d se representa así:

    - / \ + d / \ a * / \ b c

    La evaluación se hace en postorden (hojas primero, raíz al final), lo que garantiza respetar la precedencia sin paréntesis.

    ✅ Ventajas
    • Eliminan ambigüedad
    • Facilitan la evaluación semántica
    • Base para código intermedio
    • Permiten optimizaciones locales
    🔧 Usos prácticos
    • Verificación de tipos
    • Generación de cuádruplos
    • Evaluación de constantes
    • Simplificación algebraica
    📌 1.2 Acciones Semánticas de un Analizador Sintáctico

    Las acciones semánticas son fragmentos de código que se ejecutan cuando el parser reconoce una producción gramatical. Se insertan dentro de la gramática usando notación { acción }.

    /* Gramática con acción semántica */ E → E1 + T { E.tipo = verificar(E1.tipo, T.tipo, '+') } E → T { E.tipo = T.tipo } T → id { T.tipo = tablaSimbolos.buscar(id.nombre).tipo }

    Los atributos pueden ser:

    TipoDescripciónFlujo
    SintetizadosSe calculan desde los hijos hacia el padre↑ Ascendente
    HeredadosSe pasan del padre hacia los hijos↓ Descendente
    📝En parsers LR (ascendentes), las acciones semánticas se ejecutan al hacer reduce. En parsers LL (descendentes recursivos), se ejecutan durante la derivación.

    📌 1.4 Pila Semántica en un Analizador Sintáctico

    En los analizadores LR, además de la pila de estados, existe una pila semántica paralela que almacena los atributos de cada símbolo.

    Pila de estados (LR)
    • Contiene números de estado
    • Guía las acciones shift/reduce
    • Determina qué regla aplicar
    Pila semántica
    • Contiene valores/atributos
    • Se sincroniza con la pila LR
    • Acumula resultados parciales

    Ejemplo de evaluación de 3 + 4 * 2:

    PasoAcciónPila semántica
    1Shift 3[3]
    2Shift 4[3, 4]
    3Shift 2[3, 4, 2]
    4Reduce E→E*E[3, 8]
    5Reduce E→E+E[11]
    PushPopTopReduceShift
    📌 1.5 Esquema de Traducción

    Un esquema de traducción (SDT – Syntax-Directed Translation) combina reglas gramaticales con acciones semánticas embebidas. Las acciones entre { } se ejecutan en el momento en que aparecen dentro de la derivación.

    /* Esquema de traducción para expresiones aritméticas */ E → E1 + T { E.val = E1.val + T.val; gen("ADD", E1.lugar, T.lugar, nuevoTemp()) } E → E1 - T { E.val = E1.val - T.val; gen("SUB", E1.lugar, T.lugar, nuevoTemp()) } T → T1 * F { T.val = T1.val * F.val; gen("MUL", T1.lugar, F.lugar, nuevoTemp()) } T → F { T.val = F.val; T.lugar = F.lugar } F → ( E ) { F.val = E.val; F.lugar = E.lugar } F → num { F.val = num.lexval; F.lugar = num.lexval }
    Gramática atribuida (AG)
    Cuando las acciones semánticas solo aparecen al final de cada producción. Más restrictiva pero más fácil de implementar.
    Esquema de traducción (SDT)
    Las acciones pueden aparecer en cualquier posición de la producción. Más flexible, permite atributos heredados.
    📌 1.6 Generación de la Tabla de Símbolos y de Direcciones

    La tabla de símbolos es la estructura central del compilador. Almacena toda la información relevante de cada identificador declarado en el programa.

    NombreTipoÁmbitoDirecciónValor
    xintglobal0x1000-
    pifloatglobal0x10043.14
    sumafunciónglobal0x2000-
    aintsumaSP+4-
    bintsumaSP+8-
    resintsumaSP+12-

    Asignación de direcciones:

    • Variables globales: dirección fija en el segmento de datos (ej. 0x1000)
    • Variables locales: desplazamiento relativo al puntero de pila (SP + offset)
    • Parámetros: se colocan antes del frame pointer al llamar la función
    • Temporales: se asignan dinámicamente conforme se generan
    🗂️En lenguajes con ámbitos anidados (como C o Pascal), se usa una pila de tablas de símbolos: una tabla por cada bloque o función activa.
    📌 1.7 Manejo de Errores Semánticos

    Los errores semánticos no impiden que el programa tenga sintaxis válida, pero sí impiden que sea correcto lógicamente.

    Tipo de errorEjemploMensaje típico
    Variable no declaraday = 10;"y" no fue declarada
    Tipos incompatiblesint x = "hi";No se puede asignar string a int
    Redeclaraciónint x; int x;"x" ya fue declarada en este ámbito
    Argumentos incorrectossuma(1,2,3)Se esperaban 2 argumentos, se dieron 3
    Retorno incorrectoint f(){ return "a"; }Tipo de retorno no coincide
    División por cero constanteint x = 5/0;División por cero en tiempo de compilación

    Estrategias de recuperación:

    • Modo pánico: descartar tokens hasta encontrar un punto de sincronización (punto y coma, fin de bloque)
    • Corrección local: intentar corregir el error (asumir un tipo por defecto)
    • Continuar y acumular: reportar el error, continuar el análisis y reportar todos los errores al final
    🧠 Conclusión

    El análisis semántico cierra la brecha entre la forma (sintaxis) y el significado (semántica) de un programa. Gracias a la tabla de símbolos, los árboles de expresión y las acciones semánticas, el compilador puede detectar errores lógicos, gestionar tipos correctamente y preparar toda la información necesaria para la generación de código intermedio y código objeto.
