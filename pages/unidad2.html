🎯 Introducción

La generación de código intermedio transforma el árbol sintáctico anotado (resultado del análisis semántico) en una representación abstracta e independiente de la máquina destino. Esta capa intermedia tiene dos grandes ventajas:

🔄 Portabilidad
  • El mismo front-end sirve para múltiples arquitecturas
  • Solo se reemplaza el back-end por cada CPU target
  • Principio del compilador multi-plataforma (LLVM, GCC)
⚡ Optimización
  • Es más fácil optimizar código intermedio que ensamblador
  • Permite análisis globales de flujo
  • Independencia del hardware facilita transformaciones
📌 2.1 Notaciones: Prefija, Infija y Postfija

Toda expresión puede escribirse en tres notaciones distintas según la posición del operador respecto a sus operandos:

NotaciónPosición del operadorEjemplo: a + b * cVentaja
InfijaEntre operandosa + b * cLegible para humanos
Prefija (Polaca)Antes de operandos+ a * b cSin paréntesis, evaluación directa
Postfija (Polaca Inversa)Después de operandosa b c * +Ideal para máquinas de pila

Conversión de infija a postfija usando el algoritmo de pila (Shunting-yard de Dijkstra):

Expresión: (a + b) * (c - d) Paso a paso: Leer 'a' → salida: a Leer '+' → pila: [+] Leer 'b' → salida: a b Leer ')' → salida: a b + (vaciar hasta '(') Leer '*' → pila: [*] Leer 'c' → salida: a b + c Leer '-' → pila: [*, -] Leer 'd' → salida: a b + c d Fin → salida: a b + c d - * Resultado postfijo: a b + c d - *
💡La notación postfija es la más usada internamente en compiladores porque se evalúa linealmente con una pila, sin necesidad de retroceder.
📌 2.2 Representaciones de Código Intermedio
2.2.1 Notación Polaca

La notación polaca inversa (RPN) usa una pila para evaluar. Cada operando se apila; al encontrar un operador se extraen los operandos y se apila el resultado.

Código fuente: z = x + y * 2 Postfija: x y 2 * + z = Evaluación con pila: Push x → [x] Push y → [x, y] Push 2 → [x, y, 2] * → [x, y*2] + → [x + y*2] z = → z recibe el valor
2.2.2 Código P

El Código P (P-code) es un lenguaje de bajo nivel abstracto diseñado para una máquina de pila virtual. Fue popularizado por el compilador de Pascal. Sus instrucciones principales:

InstrucciónSignificado
LOD n, aCarga el valor de la variable en dirección a, nivel n
STO n, aAlmacena el tope de pila en la dirección a
LIT 0, vApila el valor literal v
OPR 0, opAplica operación aritmética/lógica
JMP 0, aSalto incondicional a dirección a
JPC 0, aSalto si el tope de pila es falso
CAL n, aLlama procedimiento en dirección a
/* Código fuente: if x > 0 then y := x * 2 */ Código P generado: LOD 0, x ; carga x LIT 0, 0 ; carga constante 0 OPR 0, 12 ; operación ">" JPC 0, fin ; si falso, salta a fin LOD 0, x ; carga x LIT 0, 2 ; carga 2 OPR 0, 4 ; operación "*" STO 0, y ; guarda en y fin:
2.2.3 Triples

En los triples, cada instrucción tiene forma (operador, arg1, arg2). El resultado NO se almacena en un temporal explícito; se referencia por el número de instrucción.

Expresión: a = b * c + b * c (0) * b c ← calcula b*c, resultado = posición (0) (1) * b c ← calcula b*c, resultado = posición (1) (2) + (0) (1) ← suma los resultados anteriores (3) = a (2) ← asigna a 'a'

⚠️ Problema: si se reordenan instrucciones durante optimización, las referencias numéricas se invalidan.

2.2.4 Cuádruplos

Los cuádruplos tienen forma (operador, arg1, arg2, resultado). El resultado es un temporal explícito, lo que los hace más robustos para optimización.

#OpArg1Arg2Resultado
0*bct1
1*bct2
2+t1t2t3
3=t3—a
Triples
  • Menos espacio (no guarda nombre de temp)
  • Difícil de reordenar
  • Referencia por posición
Cuádruplos ✅ preferidos
  • Más espacio
  • Fácil de reordenar para optimización
  • Temporales explícitos y nombrados

2.3.2 Expresiones

Cada subexpresión genera un temporal. Se evalúa el árbol en postorden:

Expresión: (a + b) * (c - d) Cuádruplos: t1 = a + b t2 = c - d t3 = t1 * t2
2.3.3 Instrucción de Asignación
Código fuente: x = a * b + c Cuádruplos: t1 = a * b t2 = t1 + c x = t2
2.3.4 Instrucciones de Control

if-else:

Código fuente: if (a > b) x = 1; else x = 2; Cuádruplos: t1 = a > b if_false t1 goto L1 x = 1 goto L2 L1: x = 2 L2: ...

while:

Código fuente: while (i < n) i = i + 1; Cuádruplos: L1: t1 = i < n if_false t1 goto L2 t2 = i + 1 i = t2 goto L1 L2: ...
2.3.5 Funciones
InstrucciónSignificado

/* int suma(int a, int b) { return a + b; } */ /* Llamada: resultado = suma(3, 4) */ param 3 param 4 resultado = call suma, 2 /* Cuerpo de suma: */ t1 = a + b return t1
2.3.6 Estructuras (arreglos y registros)

Arreglos: el acceso A[i] requiere calcular la dirección: dir = base_A + i * tamaño_elemento

/* A[i] = 5 */ t1 = i * 4 ; 4 bytes por entero t2 = &A + t1 ; dirección de A[i] *t2 = 5 ; almacena 5 en A[i]

Registros/structs: el campo se accede sumando el desplazamiento fijo al puntero base.

/* struct Persona { int edad; float salario; } p; */ /* p.salario → offset de salario = 4 bytes (después de edad) */ t1 = &p + 4 t2 = *t1 ; valor de p.salario
🧠 Conclusión

La generación de código intermedio es el corazón del compilador. Los cuádruplos son la representación más usada en la práctica porque equilibran expresividad y facilidad de optimización. Cada estructura del lenguaje fuente —desde una suma hasta una llamada a función— se traduce en una secuencia de operaciones simples de tres direcciones que cualquier back-end puede traducir a ensamblador.
