🎯 Introducción

La generación de código objeto es la fase final del compilador. Transforma el código intermedio optimizado en instrucciones específicas para una arquitectura de hardware. El resultado puede ser:

Tipo de salidaDescripciónEjemplo
Código ensambladorTexto simbólico legible, luego ensambladoarchivo .asm → .o
Código máquina relocatableBinario con referencias pendientesarchivo objeto .o
Código máquina absolutoBinario listo para ejecutararchivo .exe / ELF

Los tres problemas principales que debe resolver el generador de código son: selección de instrucciones, asignación de registros y ordenamiento de instrucciones.

📌 4.1 Registros

Los registros son la forma de almacenamiento más rápida disponible para el procesador. Están integrados directamente en la CPU y el acceso a ellos no requiere ningún ciclo de bus de memoria.

Tipos de registros (arquitectura x86-64):

RegistroTamañoUso convencional
RAX / EAX64/32 bitsAcumulador, valor de retorno de funciones
RBX / EBX64/32 bitsBase, preservado entre llamadas
RCX / ECX64/32 bitsContador en ciclos, 4° argumento
RDX / EDX64/32 bitsDatos, 3° argumento, división/mult
RSI / RDI64 bits2° y 1° argumentos de funciones
RSP64 bitsStack Pointer — apunta al tope de la pila
RBP64 bitsBase Pointer — marco de pila actual
RIP64 bitsInstruction Pointer — dirección de siguiente instrucción
R8–R1564 bitsRegistros de propósito general adicionales (x86-64)
XMM0–XMM15128 bitsPunto flotante y operaciones SIMD

Problema de asignación de registros: hay más variables temporales que registros físicos. El compilador debe decidir cuáles variables mantener en registros y cuáles "derramar" (spill) a la memoria.

Graph Coloring (Coloreo de grafo de interferencia)
El algoritmo más usado: construye un grafo donde cada nodo es una variable y hay una arista si dos variables están vivas al mismo tiempo. Si el grafo puede colorearse con k colores (k = número de registros), no hay spill. Si no, se selecciona una variable para spill a memoria.
/* Ejemplo: calcular cuáles vars están vivas simultáneamente */ t1 = a + b ← t1 nace aquí t2 = c * d ← t2 nace aquí (t1 y t2 coexisten → interferencia) t3 = t1 + t2 ← t1, t2 mueren; t3 nace x = t3 ← t3 muere Grafo de interferencia: t1 ── t2 (coexisten en instrucción 2) t1, t2 no interfieren con t3 Si hay 2 registros: t1→R1, t2→R2, t3→R1 (reutiliza R1)
📌 4.2 Lenguaje Ensamblador

El lenguaje ensamblador es la representación simbólica del código máquina. Cada instrucción ensamblador corresponde 1:1 con una instrucción de la CPU. El ensamblador (assembler) lo convierte a binario.

Instrucciones fundamentales x86-64:

InstrucciónSintaxisEfecto
MOVMOV dst, srcCopia src en dst
ADDADD dst, srcdst = dst + src
SUBSUB dst, srcdst = dst – src
IMULIMUL dst, srcdst = dst × src (enteros con signo)
IDIVIDIV divisorRAX = RDX:RAX ÷ divisor
CMPCMP a, bCalcula a–b y actualiza flags (no guarda)
JMPJMP etiquetaSalta incondicionalmente
JE / JNEJE etiquetaSalta si igual / si no igual
JL / JGJL etiquetaSalta si menor / si mayor
PUSHPUSH regColoca reg en la pila (RSP -= 8)
POPPOP regExtrae de la pila a reg (RSP += 8)
CALLCALL procGuarda dirección de retorno y salta a proc
RETRETRetorna al llamador (lee dirección de pila)
LEALEA dst, [mem]Carga la dirección de mem en dst (no el valor)
SHL / SHRSHL reg, nShift left/right n bits (mult/div por 2ⁿ)

Ejemplo completo: función factorial iterativa

; int factorial(int n) { int r=1; while(n>1) { r*=n; n--; } return r; } section .text global factorial factorial: push rbp mov rbp, rsp ; establece marco de pila mov ecx, edi ; n → ECX (primer argumento en EDI) mov eax, 1 ; r = 1 (resultado en EAX) .loop: cmp ecx, 1 ; n > 1 ? jle .done ; si no, terminar imul eax, ecx ; r = r * n dec ecx ; n = n - 1 jmp .loop ; repetir .done: pop rbp ret ; retorna EAX (valor de r)
🔧El ABI (Application Binary Interface) define las convenciones de llamada: qué registros guardan los argumentos, cuál guarda el retorno, y cuáles deben preservarse entre llamadas. En Linux x86-64 (System V ABI): argumentos en RDI, RSI, RDX, RCX, R8, R9; retorno en RAX.
📌 4.3 Lenguaje Máquina

El lenguaje máquina es la representación binaria que la CPU ejecuta directamente. Cada instrucción se codifica como una secuencia de bytes según el conjunto de instrucciones (ISA) de la arquitectura.

Estructura de una instrucción x86:

EnsambladorCódigo máquina (hex)Descripción
NOP90No operación (1 byte)
RETC3Retorno de función
MOV EAX, 5B8 05 00 00 00Carga inmediato 5 en EAX
ADD EAX, EBX01 D8Suma EBX a EAX
PUSH RBP55Empuja RBP a la pila
MOV RBP, RSP48 89 E5Copia RSP en RBP (64 bits)
JMP +10EB 0ASalto relativo +10 bytes
CALL procE8 XX XX XX XXLlama a proc (offset 32 bits)

Proceso de generación:

  • Selección de instrucción: mapear cada cuádruplo a una o más instrucciones máquina de la ISA destino
  • Resolución de etiquetas: calcular los offsets reales de los saltos y llamadas
  • Generación de código relocatable: crear el archivo .o con símbolos pendientes de enlace
  • Enlace (linker): resolver referencias entre módulos y generar el ejecutable final
    • RISC (ARM, RISC-V)
    • Instrucciones de tamaño fijo (32 bits)
    • Pocas instrucciones, muy regulares
    • Load/Store: solo estas tocan memoria
    • Muchos registros (32)
    • Más instrucciones, más simple de compilar
    CISC (x86-64)
    • Instrucciones de tamaño variable (1-15 bytes)
    • Muchas instrucciones especializadas
    • Operandos en memoria directamente
    • Pocos registros (16)
    • Menos instrucciones en código, más complejo
    📌 4.4 Administración de Memoria

    El compilador organiza el espacio de memoria del proceso en segmentos con propósitos distintos:

    Alta dirección
    Argumentos del sistema y variables de entorno
    Stack ↓
    Variables locales, parámetros, dirección de retorno, registros salvados. Crece hacia abajo.
    ↑ Heap
    Memoria dinámica (malloc/new/free). Crece hacia arriba. El programador la gestiona.
    BSS
    Variables globales y estáticas no inicializadas. Ocupan cero bytes en el binario.
    .data
    Variables globales y estáticas inicializadas. Ej: int x = 5;
    .rodata
    Datos de solo lectura: literales de cadena, constantes. Ej: "Hola"
    .text
    Instrucciones del programa (código máquina). Solo lectura y ejecución.
    Baja dirección
    Dirección 0x000000 (no mapeada, acceso = segfault)

    Marco de activación (Activation Record / Stack Frame):

    Cada llamada a función crea un frame en la pila con esta estructura:

    Dirección alta ┌─────────────────────────────┐ │ Argumentos extra (si > 6) │ ← pasados por la función llamadora ├─────────────────────────────┤ │ Dirección de retorno (RIP) │ ← guardada por CALL ├─────────────────────────────┤ ← RBP apunta aquí │ Viejo RBP (frame anterior) │ ← guardado por el prólogo ├─────────────────────────────┤ │ Variables locales │ ← asignadas por el compilador ├─────────────────────────────┤ │ Registros salvados │ ← los que el callee debe preservar └─────────────────────────────┘ ← RSP apunta aquí Dirección baja
    /* Prólogo estándar de función (x86-64) */ push rbp ; guarda el frame pointer anterior mov rbp, rsp ; establece el nuevo frame pointer sub rsp, 32 ; reserva 32 bytes para variables locales /* Epílogo estándar */ mov rsp, rbp ; restaura el stack pointer pop rbp ; restaura el frame pointer anterior ret ; regresa al llamador

    Estrategias de administración del heap:

    EstrategiaDescripciónVentaja
    First FitAsigna el primer bloque libre suficientemente grandeRápido
    Best FitAsigna el bloque libre más pequeño que sea suficienteMenos desperdicio
    Buddy SystemDivide la memoria en potencias de 2Fácil de fusionar fragmentos
    Garbage CollectionEl runtime libera automáticamente la memoria no usadaSin memory leaks
    🧠 Conclusión

    La generación de código objeto es donde el compilador toca el hardware real. Dominar esta fase implica entender la arquitectura del procesador destino: su ISA, sus registros, sus convenciones de llamada y su modelo de memoria. El compilador debe tomar decisiones críticas de asignación de registros, selección de instrucciones y gestión de la pila para producir código correcto y eficiente que la CPU pueda ejecutar directamente.
